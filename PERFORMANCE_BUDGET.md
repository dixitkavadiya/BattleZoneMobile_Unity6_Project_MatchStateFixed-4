# BattleZone Mobile - Performance Budget

This budget targets premium-looking but mobile-readable visuals. It is a planning budget, not a claim that the current prototype reaches final device performance.

## Target Device Class

- Mid-range Android device.
- Target frame rate: 60 FPS when final art is optimized and profiled on device.
- Current pass did not run on physical Android hardware, so no real-device FPS or thermal claim is made.

## Scene Budgets

- Main camera visible triangles target: 250k-450k on High, 150k-250k on Medium, under 150k on Low.
- Draw calls target: under 160 visible draw calls on Medium, under 240 on High.
- SetPass calls target: under 90 on Medium.
- Texture memory target: 350-500 MB for full match on High, lower through quality texture limits.
- Realtime lights: one directional light; additional realtime lights avoided except carefully bounded hero moments.
- Reflection probes: baked/static or low-frequency runtime probes only; no dense probe grids.
- Milestone 21 Terrain target: heightmap 129, alphamap 128, constrained detail grass, instanced Terrain rendering, grouped POIs, LOD groups on larger structures, runtime NavMesh source bounds limited to the playable island, and mobile-first terrain distances.
- Milestone 20 Terrain target: heightmap 129, alphamap 128, constrained detail grass, instanced Terrain rendering, and mobile-first terrain distances.

## LOD Rules

- Every character, vehicle, building, vegetation set, and large prop must have LOD0/LOD1/LOD2.
- LOD0 at approximately 0.25 screen-relative height.
- LOD1 at approximately 0.09.
- LOD2 at approximately 0.025.
- Use billboard/card substitutes only for distant vegetation and never for nearby interactable cover.

## URP Quality Presets

- `Low`: short shadows, reduced LOD bias, reduced texture mip budget, low particle raycast budget.
- `Medium`: default mobile target for broad testing.
- `High`: current editor smoke-test default and intended mid-range visual target.
- `Ultra`: editor/upper-device preview with longer shadows and higher LOD bias; not the default Android target.

## Material And Batching Guidelines

- Reuse material atlases by asset family.
- Avoid unique material instances on repeated props.
- Prefer static batching for buildings and props; keep dynamic batching expectations modest.
- Use object pooling for effects, tracers, impacts, and damage numbers.
- Avoid runtime allocations in visual update loops.

## Actual Editor Results For This Pass

- Unity version: `6000.5.3f1`.
- `BZ_Main` editor batch smoke test passed with VisualSlice, Milestone 17-21, Terrain, weather, probe, LOD, URP material, runtime NavMesh, terrain shader, and missing-reference validations.
- Zip archive integrity test passed after packaging.
- The Unity batch log contains one external licensing metadata line: `Access token is unavailable; failed to update`. It is not generated by BattleZone project code.
- Android device profiling was not run.

## Known Performance Risks

- The legacy runtime-generated world still contains many separate primitive/proxy objects.
- Milestone 21 adds a larger first playable island layer with terrain detail grass, many generated POI collision pieces, reflection probes, and runtime NavMesh generation; this is acceptable for editor validation but still needs real Android profiling and authored scene baking.
- Milestone 20 adds runtime Terrain detail grass, rain particles, and dynamic clouds; these are mobile-conscious but still need real-device profiling.
- Final art will need mesh combining, atlases, occlusion-friendly interiors, and baked collision.
- Vegetation density must be tuned after real-device profiling.
- True baked lightmaps and baked occlusion data are not produced by the runtime generator; final art scenes should be authored and baked in Unity before production.
- Current procedural audio and visual proxies are temporary and not memory-profiled as final assets.
